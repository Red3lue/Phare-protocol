§1ENS namespace hierarchy

Two namespaces live under the project's parent name phare.eth: one for sanctioned vessels, one for verifier agents. Both parents are wrapped in the NameWrapper and have CANNOT_UNWRAP burnt, which is the prerequisite that lets us burn child fuses below them.

What to read here

Three layers, top-to-bottom: the ENS root, Phare's two sub-roots, and the leaves. Vessel leaves are minted with a vessel-style fuse policy, verifier leaves with a lighter agent-style fuse policy. Section 4 explains the asymmetry. Vessel labels are deterministic (imo-<n>), verifier handles are derived from the last 6 hex chars of the agent's self-generated address (collision rate ≈ 1 in 16M, demo-acceptable).

§2Component interactions with the ENS layer

Lighthouse.sol is the single contract that mediates every ENS write. ReportRegistry is its sole authority for vessel writes; verifier wallets self-enroll directly. After enrollment, verifiers write their own records to PublicResolver without going through Lighthouse — a deliberate consequence of burning PCC on the agent's name.

What to read here

  • One contract owns the ENS surface. Every name write goes through Lighthouse, except for the verifier's own post-enrollment writes (which go straight to PublicResolver).
  • Two write authorities. Vessel writes are onlyRegistry: only ReportRegistry can call them. Verifier writes are permissionless: any wallet self-enrolls.
  • Operator approval is the trick. The deployer doesn't transfer the parent names to Lighthouse — it grants setApprovalForAll. That keeps the deployer key as a revocation safety net while letting Lighthouse mint children under either parent.

§3Pre-event setup — 9 transactions

One-shot bootstrap from a single deployer key. After step 9, the deployer never signs another ENS-related transaction during the demo — Lighthouse handles everything.

sequenceDiagram autonumber actor D as Deployer EOA participant ENS as ENS Manager (browser) participant NW as NameWrapper participant LHC as Lighthouse contract D->>ENS: 1. Register phare.eth (commit/reveal, ~0.005 ETH) Note over ENS,NW: Manager wraps by default → ERC-1155 D->>NW: 2. setSubnodeOwner(phare, "vessel", D, 0, max) D->>NW: 3. setSubnodeOwner(phare, "verifier", D, 0, max) D->>NW: 4. setFuses(phare.eth, CANNOT_UNWRAP=1) D->>NW: 5. setChildFuses(phare, "vessel", PCC|CANNOT_UNWRAP=65537) D->>NW: 6. setChildFuses(phare, "verifier", PCC|CANNOT_UNWRAP=65537) D->>LHC: 7. forge create ReportRegistry(orbitalAttestor) D->>LHC: 8. forge create Lighthouse(NW, resolver, vesselNode, verifierNode, RR) D->>NW: 9. setApprovalForAll(Lighthouse, true)

Why this exact ordering

  • Burn parent fuses before child fuses. setChildFuses on a child reverts unless the parent's CANNOT_UNWRAP is already burnt. That's why step 4 precedes 5 and 6.
  • Use setChildFuses, not setFuses, on the children. It lets the parent owner burn PARENT_CANNOT_CONTROL and CANNOT_UNWRAP in one go, in the right order.
  • Deploy ReportRegistry before Lighthouse. Lighthouse's reportRegistry field is immutable — it must know the registry address at construction time.
  • setApprovalForAll is one tx, not two. A single approval covers every wrapped name held by the deployer.

§4Trust-model asymmetry — vessel vs verifier

The most distinctive design choice in the ENS layer. Same registrar, mirror-image fuse policies, opposite trust assumptions.

Vessel subname

  • Owner: Lighthouse contract
  • Fuses burnt: PCC + CANNOT_TRANSFER + CANNOT_UNWRAP
  • Writable by: Lighthouse, only via ReportRegistry
  • Permanence: sealed forever to the contract
  • Why: a vessel cannot hold a wallet — its identity must be tamper-evident infrastructure curated by settled facts

Verifier subname

  • Owner: verifier principal EOA
  • Fuses burnt: PCC only
  • Writable by: the principal, directly
  • Permanence: belongs to the keeper
  • Why: a verifier is a wallet — policy / soul / lastDecision are self-curated public statements
graph TB subgraph vessel_side[Vessel subname — imo-N.vessel.phare.eth] VO[Owner: Lighthouse contract] VF["Fuses: PCC + CANNOT_TRANSFER + CANNOT_UNWRAP"] VW[Writable by: Lighthouse, via ReportRegistry only] VP[Permanence: sealed forever to the contract] end subgraph verifier_side[Verifier subname — handle.verifier.phare.eth] AO[Owner: verifier principal EOA] AF["Fuses: PCC only"] AW[Writable by: the principal, directly] AP[Permanence: belongs to the keeper] end VO --> VF --> VW --> VP AO --> AF --> AW --> AP

The PCC consequence

PARENT_CANNOT_CONTROL is the fuse that emancipates a child from its parent. Once burnt, neither the parent owner nor Lighthouse itself can revoke or rewrite the child. For the verifier, this is the design goal — the agent owns its name. For the vessel, PCC is paired with CANNOT_TRANSFER to lock the token to Lighthouse forever, so vessel identities can neither be moved nor sold.

§5Vessel — first sighting flow

A settled ReportRegistry event triggers the mint and the initial records, atomically inside the settlement callback.

sequenceDiagram autonumber actor R as Reporter PWA participant RR as ReportRegistry participant UMA as UMA OOv3 participant LH as Lighthouse participant NW as NameWrapper participant PR as PublicResolver R->>RR: submit(report, $5 USDC bond) RR->>UMA: assertTruth("Report at bzz://… is true") Note over UMA: liveness window (≈1 min for demo) UMA-->>RR: assertionResolvedCallback(id, true) RR->>RR: _onSettled RR->>LH: nameVessel(imo, swarmRef) LH->>NW: setSubnodeRecord(vesselParent, "imo-9133701",
owner=address(this), FUSES_VESSEL, max) NW-->>LH: node LH->>PR: setText(node, "vessel.imo", "9133701") LH->>PR: setText(node, "vessel.swarm.log", "bzz://…") LH-->>RR: emit VesselNamed(imo, node, ens)

What to read here

The mint and the first text records are written in the same transaction as the UMA settlement — a single atomic state transition. By the time VesselNamed is emitted, imo-9133701.vessel.phare.eth resolves in any ENS-aware browser. Subsequent sightings of the same vessel skip the mint and call recordSighting, which only touches PublicResolver.

§6Verifier — self-enrollment flow

Permissionless. Any wallet can call enrollVerifier and walks away owning a wrapped ERC-1155 with PCC burnt.

State machine context

The OpenClaw skill governs its lifecycle through four phases: init → needs-funding → needs-ens → running. This sequence is the needs-ens phase. Once enrolled, the skill enters running and only then begins polling UMA for assertions to dispute. Onboarding is gated on user chat input, not the heartbeat tick — the agent does not waste resources before it has an identity.

§7Verifier — post-dispute self-write

After PCC is burnt, the verifier owns its name outright and writes its own reasoning trail. Lighthouse is no longer in the path.

sequenceDiagram autonumber actor V as Verifier wallet participant UMA as UMA OOv3 participant RR as ReportRegistry participant SW as Swarm participant PR as PublicResolver UMA-->>V: AssertionMade event
(callback recipient = ReportRegistry) V->>SW: fetch metadata.json (Verified Fetch) V->>V: evaluate against policy → "fake" V->>SW: pin reasoning.json → bzz://reasoning V->>UMA: disputeAssertion(id, counter-bond) Note over UMA: voters resolve… UMA-->>RR: assertionResolvedCallback(id, false) V->>PR: setText(node, "verifier.lastDecision", bzz://reasoning)

Why this matters

verifier.lastDecision is a live, self-attested public reasoning trail — written by the agent itself, immediately after every dispute. Because PCC is burnt on the agent's name, no parent and no registrar can interfere with this write. The verifier's identity, policy, and decision history are all on-chain ENS records that the verifier's own key controls. This is the prize-relevant property for "ENS for AI Agents".

The split rule

  • On-chain: the discrete fields a UI needs for fast reads (sanction status, current Swarm pointer, last decision pointer).
  • On Swarm: append-only history, free-form prose, derivable counters, anything large or rapidly-changing.
  • Bridges: contenthash is the ENS-native pointer; vessel.swarm.log / verifier.swarm.log mirror the same Swarm hash as a plain text record for non-ENS tooling. Both always point at the same content.

The closing frame

Both subnames resolve in stock ENS tooling. No project-controlled API in the read path. The vessel's records are sealed to Lighthouse; the verifier's records are written by the verifier itself. The reasoning JSON pinned to Swarm is reachable from verifier.lastDecision in one click. The graph is fully browsable through ENS alone.
