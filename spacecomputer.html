1. ENS — what carries over from Hermes

Brief because you've shipped this pattern (Hermes/packages/sdk/src/ens.ts) and won the ENS prize with it. The Phare ENS work reuses your structure with a new namespace.

  • Read path: getEnsText(client, { name, key }) against any CCIP-Read-aware viem client — works transparently against NameStone. Templates: resolveAgent → resolveVessel, resolveBiomeRecords → resolveVerifier.
  • Write path: differs from Hermes — Phare uses NameStone's HTTP API (CCIP-Read off-chain resolver) instead of on-chain multicall. No gas per text record. Authority is a project API key, not the EOA.
  • Type modelling: keep the typed-record-struct pattern. Define VesselRecords + VerifierRecords with the keys listed in ENS_SPEC.md §3 / §4.
  • Multicall pattern: still useful if we ever set contenthash or resolver address on the parent phare.eth. Reuse the minimal RESOLVER_ABI for that.

Third-party imagery APIs (for Strategy B)

Planet

docs.planet.com/develop/apis/ · 3m daily imagery · API-key auth
  • Commercial; Education and Research program can give free access (apply ahead of time, not a day-of pickup)
  • Useful APIs: Data API (catalog search), Tile Service (XYZ tiles by lat/lon)
  • For Phare: query archived PSScene imagery near a coordinate, download a clip
  • Highest fidelity of the three options. Hardest auth flow.

NASA Open APIs

api.nasa.gov · Landsat 8 / MODIS · 30m resolution · instant API key
  • Free + instant signup — no review, no commercial gating
  • Useful endpoints: /planetary/earth/imagery?lat=&lon=&dim=&date= returns a Landsat 8 PNG
  • Also: GIBS tile service for NASA's processed imagery
  • Coarser resolution than Planet but the easiest path to "real-ish" imagery

SatNOGS DB

db.satnogs.org/api/ · open ground-station network · TLE + observations
  • Not an imagery source — radio observations and satellite metadata
  • Useful role: enrichment. Use TLE data to compute which imaging satellites had passes near our coordinates at our timestamp
  • Adds genuine, verifiable orbital metadata even when the image itself is mocked
  • Solar-punk aligned: community-run, open-source ground-station network

4. The plugin — design

One TypeScript module. Four JSON-RPC methods. A pluggable backend that picks between fixture lookup and third-party APIs at runtime.

JSON-RPC contract

MethodDirectionPurpose
phare.requestImagery({ imo, lat, lon, reportId })caller → pluginSynchronous: returns { taskingId }. Backend starts fetching imagery.
phare.imageReady({ taskingId, image, captureMetadata })plugin → caller (callback)Delivered when backend returns. Carries image bytes + capture metadata.
phare.requestInference({ taskingId, imageRef, verifierLogRefs[] })caller → pluginSynchronous: returns { inferenceId }. spaceTEE stub starts predicting.
phare.inferenceReady({ taskingId, prediction })plugin → caller (callback)Destination prediction with "mocked": true propagated.

Backend interface (so A and B are interchangeable)

// orbital-plugin/src/backends/interface.ts
export interface ImageryBackend {
  name: "fixture" | "nasa" | "planet";
  fetchImage(params: {
    lat: number;
    lon: number;
    when?: Date;            // capture time hint, not a hard constraint
    imo: string;
  }): Promise<{
    image: Uint8Array;       // JPEG bytes
    captureMetadata: {
      source: string;        // "phare-fixture" | "nasa-landsat-8" | "planet-psscene"
      sensor: string;
      resolution_m_per_px: number;
      captured_at: number;   // unix
      gps: [number, number];
      attribution: string;
    };
    fallback?: boolean;      // true if we returned a default
  }>;
}

The plugin's phare.requestImagery handler picks one backend at startup (env var PHARE_IMAGE_BACKEND), and that's it. All four methods stay identical regardless of which backend is active.

5. Strategy A — predetermined fixtures (PRIMARY)

Default backend. Ship this first. It removes every external dependency from the demo path. If Strategy B never lands, the project still demonstrates the full SpaceComputer integration with honest "mocked": true labelling.

Fixture backend

MOCK · primary orbital-plugin/src/backends/fixture.ts 1–2 h
Purpose
Return a JPEG from disk based on coordinate match. Pure function from (lat, lon) to bytes + metadata.
Lookup logic
1. Round (lat, lon) to one decimal place (≈ 11 km bucket).
2. Look up "34.7,33.0" in fixtures/manifest.json.
3. If miss, return fixtures/default-ocean.jpg with fallback: true in metadata.

      "captured_at": 1715165000,
      "attribution": "Reuters press gallery / cropped"
    },
    "36.5,22.7": {
      "file": "laconian-young-yong.jpg",
      "imo_hint": "9259325",
      ...
    }
  },
  "default": "default-ocean.jpg"
}
Coverage
Aim for 5–10 buckets covering the demo coordinates listed in SPEC.md §10 (Cyprus, Laconian Gulf cluster) plus a handful of plausible shadow-fleet hot zones (Black Sea exits, Singapore Strait, etc.) so judges can play with arbitrary inputs.

Fixture sourcing

  • Sentinel-2 (ESA, free, redistributable) via Sentinel Hub EO Browser — best general source for ocean scenes near any lat/lon.
  • Reuters / national-agency press galleries — for the named demo IMOs (PABLO 9133701, YOUNG YONG 9259325). Already in your SPEC.md §10 checklist.
  • MarineTraffic static photos as fallback (attribution required).
  • One stock ocean crop with watermark as the no-fixture default — must read "phare-mock-fixture · no orbital data" so demo-day judges can't confuse it with a real capture.

6. Strategy B — third-party imagery APIs (STRETCH)

Only after Strategy A is green. The backend interface (§4) means each B option is a self-contained module behind the same contract. None of the rest of the system changes when you wire a B backend in. Pick one, two, or none — graceful degradation is automatic.

B1. NASA Earth API backend

STRETCH · easiest orbital-plugin/src/backends/nasa.ts 2–3 h
Why first
Free, instant API-key signup, no commercial review. The path of least friction to "real-ish" imagery in the demo.
Endpoint
GET https://api.nasa.gov/planetary/earth/imagery?lat=34.7&lon=33.0&dim=0.1&date=2024-06-01&api_key=KEY → returns a Landsat 8 PNG.
Resolution
~30 m/px (Landsat 8). Coarser than Planet but fine for vessel-scale demonstration when paired with a coordinate marker.
Code shape
export const nasaBackend: ImageryBackend = {
  name: "nasa",
  async fetchImage({ lat, lon, when }) {
    const date = (when ?? new Date()).toISOString().slice(0, 10);
    const url  = `https://api.nasa.gov/planetary/earth/imagery`
               + `?lat=${lat}&lon=${lon}&dim=0.1&date=${date}`
               + `&api_key=${process.env.NASA_API_KEY}`;
    const res  = await fetch(url);
    if (!res.ok) throw new Error(`NASA Earth API ${res.status}`);
    const png  = new Uint8Array(await res.arrayBuffer());
    return {
      image: png,
      captureMetadata: {
        source: "nasa-landsat-8",
        sensor: "OLI",
        resolution_m_per_px: 30,
        captured_at: Math.floor(Date.parse(date) / 1000),
        gps: [lat, lon],
        attribution: "NASA Earth Imagery / Landsat 8 / public domain"
      }
    };
  }
};
Gotchas
The API can return 404 when no Landsat scene exists for that lat/lon/date. Always wrap in a try/catch and fall back to the fixture backend; the wrapping is what makes the system graceful.
Done when
Setting PHARE_IMAGE_BACKEND=nasa swaps the demo to live NASA imagery for any coordinate the API has data for, with automatic fallback otherwise.

B2. Planet Data API backend

STRETCH · richest orbital-plugin/src/backends/planet.ts 4–6 h
Why second
Highest fidelity (3 m/px daily), but the auth flow is gated. Apply to Planet's Education and Research program before the hackathon — this is not a day-0 pickup.
Two-step API call
1. Search: POST /data/v1/quick-search with a geometry filter (point or polygon around lat/lon) and item_types: ["PSScene"]. Returns a list of recent scenes.
2. Download: pick the most recent scene's thumbnail URL or the visual asset (after activation).
Auth
X-PLANETLABS-API-KEY: $PLANET_API_KEY header. API key is per-account.
Code shape
export const planetBackend: ImageryBackend = {
  name: "planet",
  async fetchImage({ lat, lon }) {
    const search = await fetch("https://api.planet.com/data/v1/quick-search", {
      method: "POST",
      headers: {
        "Content-Type": "application/json",
        "Authorization": `api-key ${process.env.PLANET_API_KEY}`,
      },
      body: JSON.stringify({
        item_types: ["PSScene"],
        filter: { type: "GeometryFilter", field_name: "geometry",
                  config: { type: "Point", coordinates: [lon, lat] } }
      }),
    });
    const { features } = await search.json();
    const scene = features?.[0];
    if (!scene) throw new Error("no Planet scene at coordinates");
    const thumb = await fetch(scene._links.thumbnail);
    const png   = new Uint8Array(await thumb.arrayBuffer());
    return {
      image: png,
      captureMetadata: {
        source: "planet-psscene",
        sensor: "PSScene",
        resolution_m_per_px: 3,
        captured_at: Math.floor(Date.parse(scene.properties.acquired) / 1000),
        gps: [lat, lon],
        attribution: `Planet Labs / scene ${scene.id}`
      }
    };
  }
};
Gotchas
Education program approval can take days. Without it you only get demo-tier access (mostly historical, no visual asset).
Thumbnails are fine for a slide; full assets require an Order API call (extra latency).
Done when
Setting PHARE_IMAGE_BACKEND=planet returns 3 m/px imagery for the demo coordinates.

B3. SatNOGS DB enrichment

STRETCH · enrichment orbital-plugin/src/enrichers/satnogs.ts 2–3 h
Special role
Not a primary image backend. SatNOGS is a community ground-station network — its DB carries real TLE data and observation logs, not satellite imagery. Use it to enrich the captureMetadata with verifiable orbital facts.
What it adds to the metadata
nearby_satellites: list of imaging satellites whose orbital ground tracks passed within N km of (lat, lon) within ±2 hours of captured_at, computed from TLE data + standard SGP4 propagation.
Output example: 
"nearby_satellites": [
  { "name": "SENTINEL-2A", "norad_id": 40697, "pass_at": 1715164800, "min_distance_km": 3.2 },
  { "name": "LANDSAT 8",  "norad_id": 39084, "pass_at": 1715166600, "min_distance_km": 18.7 }
],
"satnogs_attribution": "TLE data via db.satnogs.org"
How it works
1. Pull TLE for ~10 known imaging satellites once at startup (cache for 24h).
2. For each fetchImage call, propagate orbits with satellite.js and check for ground-track distance to (lat, lon).
3. Attach the matching list to captureMetadata.nearby_satellites.
Why this is interesting beyond gimmick
It makes the orbital corroboration metadata partially real even when the image is mocked — judges can independently verify "yes, Sentinel-2A really did pass over there at that time." That's a defensible second signal.
Dependencies
satellite.js (npm) for SGP4; db.satnogs.org/api/satellites/ + /api/transmitters/ for the satellite catalog.
Done when
Every captureMetadata includes a nearby_satellites array, regardless of which image backend is active.

Backend selector

// orbital-plugin/src/backends/selector.ts
import { fixtureBackend } from "./fixture";
import { nasaBackend }    from "./nasa";
import { planetBackend }  from "./planet";

const REGISTRY = {
  fixture: fixtureBackend,
  nasa:    nasaBackend,
  planet:  planetBackend,
};

export function pickBackend(): ImageryBackend {
  const choice = process.env.PHARE_IMAGE_BACKEND ?? "fixture";
  return REGISTRY[choice as keyof typeof REGISTRY] ?? fixtureBackend;
}

// Wrap any backend with automatic fallback to fixture on failure
export function withFallback(primary: ImageryBackend): ImageryBackend {
  return {
    name: primary.name,
    async fetchImage(p) {
      try { return await primary.fetchImage(p); }
      catch (err) {
        console.warn(`[${primary.name}] failed, falling back to fixture:`, err);
        return fixtureBackend.fetchImage(p);
      }
    }
  };
}

7. spaceTEE inference — mock stub

Independent of the imagery strategy. The TEE is mocked because spaceTEE itself is on SpaceComputer's roadmap, not shipped.

Mock spaceTEE inference

MOCK orbital-plugin/src/tee/mock-tee.ts 3–4 h
Purpose
Given an image ref + verifier-log refs, predict a destination port for the vessel.
Algorithm (rule-based, intentionally crude)
1. Read the image's capture metadata for last-known GPS.
2. Fetch the last 1–3 sightings from the vessel's vessel.swarm.log on Swarm.
3. Compute extrapolated heading.
4. Match against a static map of likely sanctioned-tanker destination ports (Novorossiysk, Tartus, Kaliningrad, Visakhapatnam, Sikka, Ningbo).
5. Score = inverse great-circle distance × heading alignment.
Output JSON
{
  "schema": "phare.tee-inference/1",
  "destination": "Novorossiysk",
  "confidence": 0.78,
  "route_candidates": [
    { "port": "Novorossiysk", "score": 0.78 },
    { "port": "Tartus",       "score": 0.41 }
  ],
  "supporting_logs": ["bzz://...", "bzz://..."],
  "image_ref": "bzz://...",
  "mocked": true
}
Critical detail
"mocked": true propagated into the Swarm-pinned JSON. Consumers (and judges) filter on it.
Done when
Given any image fixture + at least one verifier-log ref, returns a prediction in <500 ms.

8. Supporting objects (outside the plugin)

The plugin is the ecosystem contribution. These five objects glue it to Phare's contracts, ENS, and PWA.

orbital/ orchestrator service

BUILD orbital/ 3–4 h

Subscribes to ReportRegistry.Settled; calls plugin methods; pins to Swarm; signs via KMS; calls ReportRegistry.attest(). The off-chain conductor.

ReportRegistry.attest() — Solidity

BUILD contracts/src/ReportRegistry.sol 6–8 h

Verifies EIP-191 signature recovers to immutable orbitalAttestor; marks report corroborated; stores imageHash; emits OrbitallyCorroborated. OpenZeppelin ECDSA; Foundry tests for happy path, wrong signer, replay, wrong status.

Minter extension — vessel.orbital.* writer

BUILD minter/src/orbital-writer.ts 2–3 h

On OrbitallyCorroborated: write all vessel.orbital.* text records via NameStone, refresh vessel.swarm.log. Uses your existing minter scaffold.
